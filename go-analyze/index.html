This book is the complete documentation for the go − analyze program, a static analysis tool for
the Go programming language.
What does it do? go − analyze examines source code to ﬁnd diﬀerent kinds of bugs without
ever compiling or running your program. Typical bugs found include nil-pointer errors, out of
bounds array and slice indexes, and concurrency bugs, like forgetting to take or release a lock.
What are the means? go − analyze uses the technique of abstract interpretation to symbolli-
cally approximate the execution of Go code, and is built upon an extension of Hoare logic called
seperation logic to analyze the program’s heap and stack.
How far along is the project? Not very. We can parse types now and thats about it! Expect
it to be awhile.
How is this book written? This book and project take what is called a literate programming
approach to software development. The idea is to intermix prose and code in a way that best
explains the human thoughts and intentions behind a program, and not simply the low-level con-
cerns of how code is organized into folders and ﬁles. The goal is to give humans a place to start
reading your code, along with explanations every step of the way so that by the end of the literate
program text, they should understand the entire program.
How is that interweaving accomplished? We have a very short, yet custom built tool that
analyzes source code ﬁles for special comments that denote when speciﬁc, named sections begin
and end. It then takes the results of that analysis and inserts it into the book’s L
A
T
E
X ﬁles.
Unlike in a traditional literate programming system, there is no preprocessing step that spits
out the source code to the program, in fact, the entirety of the source code need not appear in the
book (though all of it should.) This gives us the advantage that someone can compile or edit the
code without knowing the homebrew literate programming system at all.

Now there’s one function, lexToken, which is the starting point for parsing any token. Parsing
a sourceﬁle is just zero or more uses of this action.
 : Deﬁne lexToken  ≡ src/Go/Lexer.hs:289-304
lexToken :: Parser TokenAndPos
lexToken = do
pos <- P.getPosition;
c <- lookAheadOneChar; -- We factor out the lookahead and pass it
-- as an argument to all the subsequent
-- functions.
t <- lexNewline c -- Specially handled for the semi-colon
-- insertion rule.
<|> lexKeywordOrIdent c
<|> lexOperatorDelimiter c
<|> lexNumberLit c
<|> lexRuneOrStringLit c;
whiteSpace; -- We consume as much whitespace as possible.
return $ TokenAndPos t pos;
We’ll return to the whitespace action at the end, because it not only consumes whitespace, but
parses and saves comments.
2.1.1 Identiﬁers
First up we have the identiﬁer token, this is used for every kind of identiﬁer in the program:
 : Identiﬁer Token  ≡ src/Go/Lexer.hs:46-48
= Ident !T.Text
Identiﬁers start with a letter, and are followed by zero or more letters or digits. As shown in
the following EBNF from the Go spec.
identifier = letter { letter | unicode digit } .
We use isLetter to correspond to letter from the EBNF spec. Note that underscores are
considered to be a letter.
 : Deﬁne isLetter  ≡ src/Go/Lexer.hs:206-209
isLetter :: Char -> Bool
isLetter c = C.isLetter c || c == ’_’
With isUnicodeDigit we’re just renaming a function from Data.Char :
 : Deﬁne isUnicodeDigit  ≡ src/Go/Lexer.hs:203-206
isUnicodeDigit :: Char -> Bool
isUnicodeDigit = C.isNumber
3
We can combine the previous two functions into a parsing action.
 : Deﬁne lexIdentiﬁer  ≡ src/Go/Lexer.hs:311-317
lexIdentifier :: Parser Token
lexIdentifier = do
c <- PC.satisfy isLetter;
cs <- P.many $ PC.satisfy (\x -> isLetter x || isUnicodeDigit x);
return (Ident $ T.pack $ c:cs);
Note that we use T.pack to pack the string, because the P.many combinator returns a list.
